# CLAUDE.md — Email Blast

## Purpose
A white-label **bulk email operations tool**. Build on-brand emails (event invites,
reminders, etc.) from blocks/templates, send them to many recipients via **Resend**, and
track delivery status. The core value: non-developers (operators) can produce on-brand
email without writing HTML.

- Operating domain (example): `https://email-blast.example.com`
- Users: a small internal team, low-frequency sending.

## Stack
- **Next.js 14 (App Router) + React 18 + TypeScript + Tailwind**
- Sending: **Resend** SDK (NDJSON streaming for per-recipient progress)
- Image storage: pluggable — local disk (default), Azure Blob, or Cloudflare R2 (see "Images")
- Auth: two modes via `brand.config.ts` `auth.mode` — **password** or **Google Workspace OAuth** — both issue an HS256 JWT session cookie
- Tests: vitest (pure logic only — renderer, session)
- Deploy: any Node host behind a reverse proxy (see `README` / `docs/setup.md`)
- **No database.** State is stored in files (see "Data storage").

## Terminology (UI)
- The thing you send = **email**; the composer preset = **template**.
- But code identifiers, directories (`templates/`), API paths, and data fields (`templateName`, etc.)
  stay as-is for data compatibility. **Only UI strings** use the new terms.

## Environment variables / secrets
**Secrets (keys) live in `.env.local`; brand (public identity) lives in `brand.config.ts`.**
For white-label setup and details see `docs/white-label.md`.
Never hardcode values into code or docs (reference names only).

| Variable | Purpose |
|---|---|
| `RESEND_EMAIL_TRACKING_API_KEY` | Sending + tracking (full-access Resend key) |
| `ACCESS_PASSWORD` | Password-mode login secret (used when `auth.mode = "password"`) |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | Google OAuth (used when `auth.mode = "google"`) |
| `AZURE_STORAGE_ACCOUNT` / `_KEY` / `_CONTAINER` | Azure Blob image upload (when `assets.provider = "azure"`) |
| `R2_*` | Cloudflare R2 credentials (when `assets.provider = "r2"`) |
| `AUTH_SESSION_SECRET` | Session JWT signing (= `JWT_SECRET`) |
| `ADMIN_EMAILS` / `APP_BASE_URL` / `SENDER_*` | (Optional) admins · app URL · compliance. Defaults come from `brand.config.ts` |

- **Auth mode, login domain, and sender domain are managed in `brand.config.ts`**
  (`auth.mode` / `auth.loginDomain` / `auth.senderDomain`), not in env. They are constants there
  so client hydration stays consistent.
- In **password** mode, the single `auth.operatorEmail` is the only user and is automatically an admin.
- In **google** mode, login is restricted to `auth.loginDomain`, and `GOOGLE_CLIENT_*` must be set.
  Register the redirect URI in your OAuth console: `{origin}/api/auth/google/callback`
  (one per origin, e.g. localhost:3001 and your deploy domain).

## Images
Two independent axes, both configured in `brand.config.ts` `assets` (credentials stay in `.env.local`).
See `docs/images.md` for the full pipeline.

- **`assets.provider`** — where uploaded images are stored:
  - `local` (default) — written to disk and served by the `/api/assets/[file]` route.
  - `azure` — Azure Blob (`lib/storage/adapters/azure.ts`).
  - `r2` — Cloudflare R2.
  - Adapters live in `lib/storage/adapters/`; switching the provider swaps the backend.
- **`assets.delivery`** — how images reach the recipient's inbox:
  - `attach` (default) — embedded as CID inline attachments via `lib/email-images.ts`.
  - `hosted` — referenced by public URL (uploaded through `/api/upload`).

## Core flows
- **Auth**: per `auth.mode` — password (`/api/auth/password`, verifies `ACCESS_PASSWORD`) or
  Google (`/api/auth/google` → Google with `hd=loginDomain` → callback domain check). Both issue an
  HS256 JWT session cookie; middleware guards every path by checking the session.
- **Email composition (blocks)**: arrange blocks in the UI → `/api/templates/render` for preview →
  `/api/templates/compose` to save `{name}.json` (spec) + `{name}.html` (rendered). Colors/fonts are
  fixed in `lib/blocks.ts`. Body width is spec.width (default 600 / wide 680). Saving a new same-name
  email returns 409 (editing uses `overwrite:true`).
- **Personalization**: `lib/personalize.ts` handles `{{name}}`/`{{firstName}}`/`{{name|default}}`/`{{email}}`.
  Both body (HTML-escaped) and subject (plain) are substituted per recipient at send time. Preview and the
  warning (`hasBlankNameRisk`) use the same module — preview must never diverge from the actual send.
- **Lists**: `/api/lists` CRUD. Members come from CSV/XLSX import (header synonym matching + a column-mapping
  confirm UI, parsed client-side) or free-text paste. At send time you can multi-select lists and add ad-hoc
  free-text recipients together.
- **Senders**: built-in + shared (admin-managed) + personal (nickname on the user's own address) + a virtual
  "my account" option (synthesized, not stored). Send validation is `isSenderAllowedFor` — an unregistered
  from is rejected with **400, no silent fallback**.
- **Admins**: env `ADMIN_EMAILS` (seed) ∪ `data/admins.json`. Decisions must use `isAdminAsync`/`canManageAsync`
  (beware: a missing `await` would make everyone an admin).
- **Sending**: `/api/send` merges lists + ad-hoc and dedupes → ConfirmSendModal (3-pane, the `?built=`
  token-preserving HTML is substituted per recipient on the client) → 1:1 send via Resend → NDJSON stream
  reports progress/success/failure → records accumulate in `data/sends/{id}.json`. Sending continues even if
  the client disconnects, and results remain in the file.
- **Tracking (history)**: `/api/sends` returns sends/*.json in reverse-chronological order. The sidebar polls
  every 15s (only while open). The detail modal uses `?withStatus=1` to fetch live Resend status
  (delivered/opened/bounced…) with an N+1 fetch + 5s cache.
- **Auto-cleanup**: on server boot (`instrumentation.ts`) and on `/api/sends` GET (5-minute throttle), stale
  `running` sends are marked `aborted`.

## Data storage
- **`templates/`**: **production is the source of truth** (created in the operations UI). Back these up from
  the server before deploys so overwrites don't lose them.
- **`data/lists/`, `data/sends/`, `data/logos.json`, `data/from.json`, `data/admins.json`, `data/suppression.json`**:
  whitelisted in `.gitignore` so they can be backed up to version control. Sync production → local before
  committing so a disk failure on the host doesn't lose operational data. If dev-time sends linger in the
  working tree, the next backup of production data overwrites them to keep things consistent.
- **`data/sent-ids.json.legacy`**: remnant of the old tracking structure. Unused. Ignored via `.gitignore`.

## Version control / deployment
- Any Node host behind a reverse proxy with automatic HTTPS works. The reverse proxy terminates TLS and
  proxies to the Next.js process (run via your process manager of choice).
- `.env.local` is gitignored — provision it on the host separately. Secrets never live in version control.
- For setup and production deployment steps see `docs/setup.md` and `docs/production.md`.


## Conventions
- Code, comments, commit messages: English.
- Present a plan and get confirmation before creating/modifying files, installing, or deploying. Never hardcode secrets.
- All email body text is HTML-escaped in `lib/blocks.ts`; only http/https/mailto URLs are allowed.
- Middleware runs on the Edge runtime → no Node-only modules (JWT is implemented directly with Web Crypto).
- Preview iframes are always `sandbox="allow-same-origin"` — arbitrary HTML/scripts must not touch the parent page.
- If `AUTH_SESSION_SECRET` is missing and `NODE_ENV=production`, throw immediately (no silent dev-key fallback).
